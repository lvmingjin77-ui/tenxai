# Q2 方向A：Agent驱动的自动化特征工程系统

## 1.系统架构

```text
                ┌─────────────────────────────────────────────┐
                │           FeatureAgent（Orchestrator）        │
                │        DeepSeek Chat + tool calling          │
                └──────┬───────────┬──────────┬───────────────┘
                       │           │          │
              ┌────────▼──┐  ┌─────▼───┐  ┌──▼──────────┐
              │exec_python│  │  state  │  │log_decision │
              │  执行代码  │  │ 共享状态 │  │  决策日志   │
              └────────┬──┘  └────┬────┘  └─────────────┘
                       │          │
       ┌───────────────▼──────────▼────────────────────────┐
       │                四阶段Agent Workflow                │
       │ Diagnose → Clean → Evaluate → Select              │
       └───────────────────────────────────────────────────┘
                       │
       ┌───────────────▼───────────────────────────────────┐
       │             LightGBM最终验证与报告                 │
       │ AUC / Precision / Recall / F1 + ROC + 混淆矩阵     │
       └───────────────────────────────────────────────────┘
```

这个系统不是只让LLM给建议，而是把它放进一个可执行工作流里。Agent读取任务说明后，会调用`exec_python`生成并执行Python代码，再根据执行输出继续下一步；中间结果都写进`state`，重要决策通过`log_decision`落日志。

## 2.工作流设计

| 阶段 | 触发方式 | Agent核心行为 | 主要输出 |
|------|---------|--------------|---------|
| Diagnose | `agent.run(DIAGNOSE)` | 生成代码诊断300个特征的缺失、分布、异常值与缺失模式 | `state['diagnose_report']` |
| Clean | `agent.run(CLEAN)` | 根据诊断结果决定缺失值填充、异常值截断、截面归一化策略 | `state['df_clean']` |
| Evaluate | 先向量化预计算，再`agent.run(EVALUATE)` | 审查IC/AUC/单调性结果，给出筛选阈值和去冗余建议 | `state['eval_df']`、`state['eval_summary']` |
| Select | `agent.run(SELECT)` | 执行四轮漏斗筛选，输出最终Top50特征 | `state['top50']`、`state['top50_scores']`、`state['select_log']` |

这里我把Evaluate拆成两层：重计算部分由确定性Python完成，判断与策略建议由Agent完成。原因很简单，300个特征乘上全部训练日期的IC批量计算更适合向量化；如果全交给Agent逐步生成循环代码，速度会明显变慢，也更容易出错。

## 3.完整Prompt归档

### 3.1 System Prompt

```text
你是一名量化研究员，负责金融时序数据的特征工程。
通过 exec_python 工具执行Python代码，通过 log_decision 记录关键决策。
规则：
1. 所有数据通过 state 字典访问（state['df'], state['x_cols'] 等）
2. 严禁数据泄漏：统计量只在 state['train_dates'] 对应的行上计算
3. 代码出错自行修复，重试不超过3次
4. 关键结果写回 state（如 state['diagnose_report'] = df）
```

### 3.2 Tool定义

```text
工具1：exec_python
- 作用：执行Python代码
- 输入：{"code": "<python code>"}
- 约束：共享命名空间为state、np、pd、stats

工具2：log_decision
- 作用：记录关键决策
- 输入：
  {
    "phase": "...",
    "feature_group": "...",
    "action": "...",
    "rationale": "..."
  }
```

### 3.3 Phase Prompt

#### Phase 1：DIAGNOSE

```text
当前共享状态：
- state['df']: 原始面板数据 (81046, 321)，含X1~X300特征
- state['x_cols']: 特征名列表（300个）
- state['target']: 目标变量 'Y7'

任务：对全部300个X特征做全面诊断，将结果保存到 state['diagnose_report']。

请生成代码（代码中可 import 任何需要的库）完成：
1. 对每个特征计算：null_pct, zero_pct, mean, std, skew, kurt, outlier_pct（IQR法）
2. 检测缺失模式：缺失行索引完全一致的特征归为同一 null_group（用frozenset hash标记）
3. 打印摘要（各null_group特征数，分布类型统计：|skew|<1为正态，1~5为偏态，>5为重尾）
4. 将DataFrame保存到 state['diagnose_report']

数据通过 state['df'] 访问，结果写回 state。
```

#### Phase 2：CLEAN

```text
当前共享状态：
- state['df']: 原始数据
- state['diagnose_report']: 诊断报告（包含null_pct, skew等列）
- state['x_cols']: 特征名列表
- state['train_dates']: 训练集日期（set类型）

任务：清理特征，严防数据泄漏（所有统计量只在训练集上计算）。

请按顺序执行：
1. 缺失值填充：
   - 先排序 df，对每个 underlying 做 ffill（前向填充）
   - 剩余缺失：用训练集中该 trade_date 的截面中位数填充
2. 异常值 clip（只用训练集统计量）：
   - |skew| > 5 的特征：用训练集 [1%, 99%] 分位截断
   - |skew| <= 5：用训练集 mean ± 3*std 截断
3. 截面 rank 归一化：每个 trade_date 内，所有 x_cols 做 rank(pct=True) → [0,1]
4. 将清理结果保存到 state['df_clean']
5. 打印每步处理的特征数和用时

用 log_decision 记录各类处理的决策理由。
```

#### Phase 3：EVALUATE

```text
当前共享状态：
- state['eval_df']: 已计算完成的特征评估表（columns: feature, ic_mean, ic_std, ic_ir, auc, monotonicity）
- state['corr_matrix']: 训练集上特征间 Spearman 相关矩阵

任务：分析评估结果，输出总体判断和筛选策略建议。

请：
1. 打印 eval_df 的 describe()
2. 统计 |ic_mean| 分布：>0.05, >0.03, 0.01~0.03, <0.01 的特征数
3. 分析相关矩阵：|corr|>0.8 的特征对数
4. 给出综合判断：
   - 信号质量总体评价
   - 推荐的 IC 过滤阈值
   - 去冗余策略建议
5. 将总结文字保存到 state['eval_summary']
```

#### Phase 4：SELECT

```text
当前共享状态：
- state['eval_df']: 特征评估表（feature, ic_mean, ic_std, ic_ir, auc, monotonicity）
- state['corr_matrix']: 相关矩阵（index/columns 为特征名）
- state['eval_summary']: 你的评估总结

任务：多轮漏斗筛选，从300个特征中选出最优50个。

请逐轮执行并打印每轮特征数：
- Round 1 基础过滤：删除 (|ic_mean| < 0.005 AND auc < 0.51) 的特征（两条件同时满足才删）
- Round 2 综合评分：score = 0.4*rank(|ic_mean|) + 0.3*rank(|ic_ir|) + 0.3*rank(auc)，取 Top 100
- Round 3 去冗余：对 Round 2 剩余特征，|corr| > 0.8 的聚为一组，每组只保留 score 最高的
- Round 4 最终截断：按 score 降序取 Top 50

关键约束（必须满足）：
1) top50 必须是 score 从高到低的前50个（禁止按升序取分数）
2) top50 中每个特征都必须来自 eval_df['feature']
3) top50 必须去重后数量=50
4) 输出并检查质量：
   - top50 的 mean(|ic_mean|) 应明显高于全集均值
   - top50 与 |ic_mean| 前100特征应有明显重合
若不满足，请自行修正后再写回 state。

保存：
- state['top50']: list，最终50个特征名
- state['select_log']: dict，{'r1': N, 'r2': N, 'r3': N, 'r4': N}
- state['top50_scores']: DataFrame，含 feature, ic_mean, ic_ir, auc, score

打印最终 Top 50 及核心指标。
```

## 4.目标标签与任务定义

我选的是`Y7`。原因是它在三分类标签里分布最均衡，`-1 / 0 / +1`大致是`34.1% / 32.0% / 33.9%`，更适合做稳定评估。最终验证模型时，我去掉`Y7==0`的中性样本，只保留`+1`和`-1`，把问题转成二分类，再计算AUC、Precision、Recall、F1。

## 5.关键实现取舍

### 5.1 截面rank归一化

这是整个方案里最关键的一步：

- 每个`trade_date`内，对全部标的的特征做`rank(pct=True)`
- 归一化后值域稳定在`[0,1]`
- 能直接消除不同特征之间的量纲差异
- 对极端值更稳，后续IC与相关性分析更可靠

### 5.2 Agent与确定性计算分层

不是所有事情都让Agent做。像批量IC、AUC、相关矩阵这种大规模计算，由确定性Python完成；Agent更适合负责策略判断、阈值建议、清理方案选择和筛选流程控制。这样可以兼顾自动化和稳定性。

## 6.异常处理机制

| 异常类型 | 处理方式 |
|---------|---------|
| Agent生成的代码执行报错 | `exec_python`会捕获traceback并把错误文本回传给模型，模型继续修复 |
| Agent未写入关键状态 | Notebook每个阶段后都做fallback检查，例如缺`diagnose_report`或`df_clean`时执行固定规则兜底 |
| 工具调用次数过多 | `FeatureAgent.run()`设置`max_iter=25`，超过后返回`(max_iter)`，避免无限循环 |
| 特征评估边界问题 | `feature_auc`和`group_monotonicity`对类别单一等情况做保护，避免异常中断 |

这些机制都能在Notebook输出里看到对应痕迹，比如有代码报错后的修复过程，也有fallback兜底日志。

## 7.数据泄漏防护

- 时间切分严格按日期做，前70%日期训练，后30%日期测试
- 缺失值填充里的截面中位数，只允许从训练集日期统计
- Winsorize分位数和`mean ± 3*std`截断阈值，只允许从训练集统计
- IC、AUC、单调性、相关矩阵全部只在训练集上计算
- Notebook最后还有单独的数据泄漏检测输出，对训练集和测试集的时间边界、日期交叉、IC量级做核验
