# 笔试题1：Q1说明

## 1.先说这个项目是干什么的

Q1是我从eggy里单独拆出来的一小块，只保留美股日频研究和回测。

eggy.png这张图就是原平台里美股页的大概样子。Q1没想把整个平台复刻一遍，而是把里面最核心、最能说明问题的那条链路单独拎出来，做成一个本地就能跑的demo。

如果看Q1.png，那张图对应的是Q1跑完一次回测之后的界面，能直接看到指标、净值曲线、调仓记录和交易流水。

我想保留的东西主要就三件：

1.真实历史数据，不用手搓样例。
2.单Agent做研究判断，不是假装有Agent，实际全靠规则。
3.回测链路完整，能看到结果，也能看到每次调仓时Agent说了什么。

所以Q1最后长成了现在这样：前端是一个很轻的静态页面，后端是Python本地服务，数据存在SQLite里。打开页面后能改参数、跑回测、看净值、看调仓记录、看成交流水。

---

## 2.我在里面做了什么

这部分我自己做得比较完整。

先是研究和回测主链路。候选股票怎么打分，市场环境怎么判断，什么时候该偏进攻、什么时候该多留现金，这些规则是我写的。候选包怎么整理给模型、模型输出什么字段、哪些字段必须校验、哪些情况要直接报错，这块也是我定的。

回测本身也是我串起来的。系统按交易日往前走，当天收盘后出判断，下一交易日开盘调仓，盘中再从开盘滚到收盘，最后更新净值、回撤和交易记录。

为了让它能单独交付，我还把工程做了裁剪：

- 后端不用Flask或FastAPI，直接用标准库起HTTP服务。
- 前端不用框架，直接写原生HTML、CSS、JS。
- 图不用第三方图表库，净值曲线直接用SVG画。
- 页面上只保留回测真的会用到的参数，不塞一堆花里胡哨的配置。

一句话说，就是把原来平台里比较重的部分拿掉，只留下能把这条美股单Agent回测链路讲清楚、跑起来、看明白的部分。

## 3.Agent目标

这套系统里的Agent，不是拿来做“全自动交易员”的，而是负责每天调仓前最后那层研究判断。

它要完成的任务其实很具体：面对当天筛出来的候选股，结合市场状态、个股最近走势、财务信息和事件上下文，判断每只票当前更适合开仓、继续持有、减仓，还是先别碰。

输入主要有四类：

- 市场上下文，比如SPY近20日和60日表现、市场宽度、平均成交额、当前偏进攻还是偏保守
- 候选股票清单，以及每只票的综合打分
- 每只候选股的结构化上下文，包括价格表现、财务指标、近期事件、未来21天内已知事件
- 当前持仓权重，方便模型区分“新开仓”和“继续拿着”

输出是一组research card。每只候选股都必须对应一张卡，里面至少要有：

- symbol
- action
- stance
- score
- confidence
- summary
- thesis
- risks
- invalidation
- lead_event_type

我这里把成功标准也定得比较死，不让模型含糊过去：

- 候选股不能漏
- action只能是enter、hold、trim、reject
- score和confidence必须落在0到1之间
- 如果某只票落在earnings_window里，就不能给新的enter
- 返回必须是能被程序直接解析的JSON

这部分其实就是整条链路最像Agent的地方。前面选候选、后面落仓都还是规则，只有这一步是模型做判断。

## 4.整体流程

Q1的回测流程不是“全都交给模型”。更准确地说，是一条“规则先筛一遍，Agent再做研究判断，最后还是规则落仓”的流程。

大概分三步。

### 第一步：先用规则生成候选

每天到调仓点，系统先看股票池里每只票最近一段时间的表现，再结合财务和事件做一个综合分数。

这里不是瞎打分，主要看几类东西：

- 相对SPY的20日、60日强弱
- 质量，比如ROE、毛利率、经营现金流比率
- 成长，比如营收增速、净利增速
- 估值，比如PE和PB
- 最近事件，比如新闻、披露、财报附近事件
- 波动率，太抖的票会被扣分

代码里对应的是这一段：

    total_score = (
        0.26 * (stock_20d - benchmark_20d)
        + 0.20 * (stock_60d - benchmark_60d)
        + 0.18 * quality_score
        + 0.14 * growth_score
        + 0.10 * value_score
        + 0.12 * event_score
        - 0.08 * volatility_20d
    )

另外系统还会顺手判断市场状态。比如：

- SPY20日涨得比较顺，市场宽度也够，就当成risk_on
- SPY20日跌得比较多，或者60日走势本来就弱，就当成defensive
- 其余情况算selective

这个状态会直接影响后面的现金比例。

### 第二步：单Agent做underwriting

候选股和市场上下文准备好以后，才会调LLM。

我这里不是让模型随便自由发挥，而是把所有候选都打成结构化JSON发过去。模型要做的事很明确：对每只候选给一张research card，说明当前应该：

- enter
- hold
- trim
- reject

同时还要给出：

- score
- confidence
- summary
- thesis
- risks
- invalidation
- lead_event_type

输出格式也卡得很死，只收JSON，不收markdown，不收散文。

### 第三步：规则落仓

模型给完判断以后，并不是它想买多少就买多少。

我这里把“研究判断”和“仓位落地”故意拆开了。模型只负责告诉我这只票值不值得上、把握多大、理由是什么。至于最后买多少，还是规则算：

- 只保留enter、hold、trim
- 用score乘confidence算原始权重
- 叠加cash_bias
- 再卡max_positions
- 再卡max_position_weight

这样做的好处很直接：方向可以参考模型，但仓位风险还是由程序兜底，不会因为模型一时兴奋就满仓乱冲。

## 5.为什么我会这么设计

这个设计其实不是为了“显得智能”，而是为了让系统更稳。

如果全靠规则，很多事件驱动场景会太硬。比如同样是新闻，有的是真的催化，有的只是噪音，纯规则不太好分。

但如果全靠模型，又很容易出两个问题：

1.格式不稳，返回内容机器吃不进去。
2.结论飘，同一天多跑两次都可能变。

所以我最后选的是中间路线。

- 规则负责把候选池收窄，把明显不靠谱的先挡掉。
- 模型负责做最后一层研究判断。
- 组合层再用规则把风险卡住。

这套思路在我自己看，是比较务实的，不会为了Agent两个字把简单问题故意搞复杂。

## 6.Prompt我怎么写的

这块我踩过不少坑。

一开始我也试过那种很自然语言的写法，比如“请分析下面这些股票并给投资建议”。结果模型回来的东西五花八门，有时候是表格，有时候是几段话，有时候还在JSON外面套代码块。人能看，程序没法接。

后来我就不跟它客气了，直接把user prompt做成一个json.dumps之后的大JSON，里面明确放：

- trade_date
- market_context
- current_weights
- candidate_packets
- required_json_schema
- rules

System prompt也尽量短，只强调三件事：

1.只能在给定候选里工作
2.只能基于给定证据判断
3.只输出JSON

这套改完以后，稳定性确实好了很多。

如果要放一段最关键的模板，我会放下面这两段。

System大意是：

    You are the underwriting phase of a single-agent US equity research workflow.
    This is a long-only backtest system.
    Work strictly inside the provided candidate set and point-in-time evidence.
    Return structured JSON only.
    Do not invent data and do not emit markdown.

User侧不是散文，而是一整段结构化JSON，大概长这样：

    {
      "phase":"candidate_underwriting",
      "trade_date":"2023-03-15",
      "market_context":{
        "regime_label":"selective",
        "market_view":"Opportunity exists, but leadership is uneven.",
        "benchmark_20d_return":0.0312,
        "benchmark_60d_return":0.0841,
        "breadth":0.5,
        "average_turnover":5230000000.0,
        "cash_bias":0.18
      },
      "current_weights":{"AAPL":0.21},
      "candidate_packets":[...],
      "required_json_schema":{
        "research_cards":[
          {
            "symbol":"candidate symbol",
            "action":"enter|hold|trim|reject",
            "stance":"high_conviction|standard|monitor",
            "score":"0..1 float",
            "confidence":"0..1 float",
            "horizon_days":"integer",
            "summary":"one concise sentence",
            "thesis":"short paragraph grounded in the packet",
            "risks":["list of specific risks"],
            "invalidation":"single explicit invalidation condition",
            "lead_event_type":"event type or empty string"
          }
        ]
      },
      "rules":[
        "Underwrite every provided candidate; do not omit symbols.",
        "If blocked_reason is earnings_window, do not use enter for a fresh position.",
        "Return strict JSON only."
      ]
    }

我现在把温度压在0.05，max_tokens用2400，并且要求response_format={"type":"json_object"}。这些都不是为了追求模型“更聪明”，而是为了让它少跑偏。

## 7.我碰到的几个实际问题

这部分我觉得比“我做了什么”更能说明我是不是真的自己做过。

### 1）返回格式不老实

最常见的问题就是嘴上说返回JSON，实际上还是包一层markdown，或者后面多跟一句废话。

所以我没有直接对整段响应json.loads，而是先从第一个{找到最后一个}，把中间那段抠出来再解析。

### 2）会漏票

候选明明有四只，模型有时候只回三张card，剩下一只直接当没看见。

这个不能忍，所以我这里会做symbol集合对比。只要缺一只，直接抛错，不让它混过去。

### 3）头部股票容易被“记忆”带跑

像NVDA这种票，模型训练里见得太多了，很容易不看我喂进去的数据，直接按印象给很乐观的结论。

这事没法彻底杜绝，只能靠prompt里反复强调“只看当前包里的证据”，再把上下文给全一点。

### 4）分数会飘

score和confidence有时候很保守，有时候又一股脑全冲到很高。这个如果直接拿来分仓，组合就会乱。

所以我一直坚持：模型给方向，规则定仓位。

## 8.验证与迭代

这块我实际做过，不只是嘴上说“我会优化”。

先说验证。我看结果主要看四类东西：

- 总收益和年化收益
- 最大回撤
- Sharpe
- 相对SPY的超额收益

除了最后的收益指标，我还会看过程是不是站得住。比如：

- 调仓记录里每次选出来的票有没有明显违背当时市场状态
- research card里的结论和事件上下文是不是对得上
- 交易次数是不是离谱，避免模型一会儿全换一遍

迭代上我主要动过三块。

第一块是Prompt形式。最早我用自然语言写法，返回格式很飘；后来改成大JSON，把schema和rules一起塞进去，稳定性提升很明显。

第二块是参数。我把温度降到了0.05，不再让同一天反复调用时结论乱跳。max_tokens也给够，避免卡片写到一半被截断。

第三块是兜底策略。模型输出再像回事，我也不会让它直接决定仓位。组合层一直保留上限控制和现金偏好，这样即使某次输出不理想，也不会把回测结果一下带歪。

如果从“最小可用版本”角度看，Q1现在这个形态就是我自己认为合适的MVP：能读真实数据，能调模型，能跑回测，能看中间过程，但工程又没有重到不适合单独交付。

## 10.怎么跑

建议先把backend/.env里的API配置补全，再生成数据库，最后启动服务。

先配.env。

至少要把这几个字段填好：

- DEEPSEEK_API_KEY
- DEEPSEEK_BASE_URL
- DEEPSEEK_MODEL
- LOCAL_RESEARCH_CACHE_DIR

先生成数据库：

    cd Q1/backend
    python3 -m pip install -r requirements.txt
    python3 generate_dataset.py --source local_cache

生成后的数据库文件在：

    Q1/backend/data/q1_market.db

再启动回测服务：

    cd Q1/backend
    python3 server.py

默认地址：

    http://127.0.0.1:8010

依赖也不是空的，当前要用到：

- requests
- pandas
- pyarrow
- yfinance

LLM配置走backend/.env。

## 12.证据区

为了不把正文写得太像流水账，我把一些能直接核对的东西放在最后。

### 证据1：数据库生成命令

    cd Q1/backend
    python3 -m pip install -r requirements.txt
    python3 generate_dataset.py --source local_cache

生成后的数据库文件：

    Q1/backend/data/q1_market.db

### 证据2：服务启动命令

    cd Q1/backend
    python3 server.py

默认地址：

    http://127.0.0.1:8010

### 证据3：模型调用关键配置

我现在实际用的是下面这组配置：

- 温度是0.05
- max_tokens是2400
- response_format要求json_object
- 超时是90秒

这组配置的目的不是追求花哨，而是尽量把输出压稳。

### 证据4：候选打分权重

候选不是随便挑的，核心打分就是这一套：

    total_score =
      0.26 * 相对20日动量
      + 0.20 * 相对60日动量
      + 0.18 * 质量分
      + 0.14 * 成长分
      + 0.10 * 估值分
      + 0.12 * 事件分
      - 0.08 * 20日波动率

### 证据5：市场状态阈值

- SPY20日收益大于8%，而且市场宽度大于等于65%，记成risk_on，cash_bias是8%
- SPY20日收益小于-4%，或者60日收益为负，记成defensive，cash_bias是32%
- 其他情况记成selective，cash_bias是18%

### 证据6：我实际处理过的报错类型

我在这条链路里重点兜过几类错：

- LLM响应里没有research_cards
- 候选有四只，但只回了三张card
- JSON外面套了一层markdown
- 请求被限流，直接回429
- 返回内容里根本找不到完整JSON对象

这些情况我没有放过，而是都在解析阶段做了硬校验。

### 证据7：Prompt迭代方向

我至少做过两版明显不同的Prompt。

第一版是自然语言写法，大概就是“请分析下面这些股票并给投资建议”。这个版本最大的问题是返回格式飘，程序没法稳定接。

第二版我把User部分改成大JSON，把schema和rules直接写进去。这个版本最大的变化不是“模型更聪明了”，而是输出明显更稳，解析成功率高很多。

### 证据8：当前默认数据集

现在页面默认围绕这份数据集在跑：

- dataset_id是hetero_trade_mas_cache_us_v1
- 股票池是AAPL、MSFT、NVDA、TSLA
- 基准是SPY
- 当前演示区间是2022-10-05到2023-06-10

这几项都能直接从项目里的数据库和页面上核对出来。
